What is VISHNU ?

VISHNU offers to end-users the capability of running computation tasks (jobs) on remote servers through a simple interface that offers performance and interoperability with different kind of batch scheduler softwares running locally on the servers. VISHNU also offers files management services to create, modify and transfer files on remote hosts using secure transfer protocols. The VISHNU system is also capable of monitoring the performance of the servers and to recover automatically from server failures.

The VISHNU software consists of a client part that provides both command-line executables and C++ shared libraries for integration into an application, and a server part that is based on the DIET middleware for remote invokation of distributed services. Other user interfaces are provided on top of the C++ API using SWIG wrappers: currently it is available as Python module and Web services server.

VISHNU is a modular system where each module can be deployed individually on a server to provide a subset of all VISHNU services. For example the TMS module can be deployed on one server for providing jobs handling services, while other VISHNU services (user mgmt, files mgmt, etc.) are served from another host. This allows a better performance by reducing the load on each server.

Requirements

VISHNU can be installed on Linux systems only (preferably Debian distributions), and requires the following software:

  • Boost C++ libraries (1.46.1 or greater)

  • OmniORB (4.1.4 or greater)

  • EMF4CPP (0.0.2 patched - see debian packages below - or greater)

  • SSH

  • PostGreSQL (8.0 or greater) or MySQL (5.0 or greater)

  • DIET version 2.8 + the log service module asociated

  • On all hosts providing task mgmt services (TMS module):

    • Torque API v3.2.6

    • IBM LoadLeveler 3.5.1.14

    • SLURM v2.2.1 (ou 2.2.7)

    • LSF 7.0.6.134609

    • OR Grid Engine 2011.11

Launch the VISHNU platform

VISHNU is a distributed system and can therefore be installed on several systems as well as on a single system. When using several systems, the choice of which services to deploy on each system is dependent on the characteristics of the systems and of the network.

Here is the list of processes that must be started to deploy all VISHNU services (see also the “VISHNU Administrator's guide” in the Documentation section):

  1. Database instance: either a MySQL or PostgreSQL database

  2. CORBA OmniNames process (omniNames)

  3. DIET Master agent process (dietAgent)

  4. DIET LogCentral process (LogCentral)

  5. All VISHNU SeD processes (umssed, tmssed, imssed, fmssed)

Glossary

  • Computing resource: a server that is used to process batch jobs. This definition covers supercomputers, the cluster gateway servers and the cluster processing nodes.

  • Host: represents a physical client or server computer system.

  • Middleware: an application that sits "in the middle" between application software that may be working on different operating systems. It is similar to the middle layer of a three-tier single system architecture, except that it is stretched across multiple systems or applications. [source: Wikipedia]

  • POSIX: international standard that defines an API to access a filesystem.

  • Storage resource: a server that is used to store data as files. This definition covers the user desktops or laptops and all unix servers.

  • SysFera-DS: open-source middleware developped by SysFera.

  • SysFera-DS Agent: SysFera-DS process running permanently (or daemon).

  • VISHNU system: set of all elements (software, hardware, data) that compose a single instance of the VISHNU application and that work together to provide VISHNU services to the users.
